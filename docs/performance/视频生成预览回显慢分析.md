# 视频生成成功后预览回显慢 — 分析与优化

> 状态：分析已落地；**P0（进度展示）已实施**  
> 日期：2026-07-15  
> 相关：[图片视频加载链路与优化方案](./图片视频加载链路与优化方案.md)

---

## 1. 现象

用户反馈：视频工作室生成 **成功后**，预览区回显偏慢（长时间进度到 100% / 黑屏，首帧迟迟不出）。

---

## 2. 成功 → 预览链路（现状）

```text
提交 IMAGE2VIDEO
  → Worker 轮询上游
  → 上游成功（progress 易被写成 100，status 仍为 running）
  → materialize() 下载整段 MP4
  → OSS putBuffer()
  → status=succeeded, progress=100
  → 前端轮询发现成功（约 2s 间隔）
  → VideoPreviewPanel <video src=签名 OSS URL preload=metadata> 无 poster
  → 浏览器拉元数据 / 首帧后才可视
```

要点：

| 环节 | 行为 |
|------|------|
| 完成感知 | 前端 `waitForGenerationJob` 每 **2s** 轮询；会话消息约 **2.5s**；SSE 接口存在但未用 |
| 对外 URL | 成功详情为 **24h 签名 OSS MP4**；视频 `thumbnailUrl` 为 **null** |
| 预览组件 | `<video>` **无 poster**，仅 `preload="metadata"` |
| 对比图片 | 有缩略图路径；提交后不必一直 await 同一套阻塞体感 |

---

## 3. 根因（按影响排序）

1. **进度过早到 100%**  
   上游状态 success 时 `progressForStatus` → `100`，视频 Worker 的 `onProgress` **未像图片那样 clamp**，直接写入 DB。此时本地仍在下载/上传，用户误以为已可预览。

2. **无视频封面 / poster**  
   成功后主预览与历史条多为完整视频 URL + 图标，首帧依赖浏览器解析大 MP4，易长时间黑屏。

3. **轮询粒度 ~2s**  
   真实 `succeeded` 后最多再等一轮轮询才刷新结果。

4. **相对图片链路更「重」**  
   须等任务真正 succeeded + 整段视频可访问；非「先下完整 blob 再挂载播放器」。

---

## 4. 优化 backlog

| 优先级 | 项 | 改动量 | 风险 | 效果 |
|--------|----|--------|------|------|
| **P0 ✅** | 进度展示不要在收尾前到 100%（运行中 cap，仅 `succeeded` 为 100） | 小 | 低 | 减少「以为好了」的错觉；体感更诚实 |
| P1 | 接入已有 SSE 或缩短轮询（2s → ~1s） | 小～中 | 低～中 | 成功后发现延迟减少约 1～2s |
| P2 | 视频 thumbnail / `<video poster>` | 中 | 中 | **黑屏体感改善最大** |
| P3 | 成功后临时用供应商 URL 先播再回退 OSS | 中偏大 | 较高 | 可能更快，与「必须落 OSS」策略需一并设计 |

**本期已实施：P0。**

### P0 落地（代码）

- `apps/api/src/modules/generation/processors/image2video.processor.ts`  
  - `onProgress`：运行中上限 95，单调递增  
  - 上游返回后 → 92（开始落盘）；上传完成 → 95；`succeeded` → 100  
- `apps/api/src/modules/generation/generation-provider.client.ts`  
  - `progressForStatus`：上游 success 类状态 → **95**（不再直接 100）

---

## 5. P0 设计

### 约定

- `status === "running"`：进度上限 **95**（上游已成功也视为「生成完成、本地收尾中」）
- `status === "succeeded"` / `"failed"`：进度 **100**（任务终态）

### 非目标（P0）

- 不缩短 materialize / OSS 实际上传时间  
- 不改前端预览组件、不新增封面

---

## 6. 验证建议

1. 发起一条较慢的视频任务，观察历史条进度：上游完成后应停在 **≤95%**，本地落库成功后再到 **100%** 且预览出现。  
2. 失败任务终态仍为 `progress: 100` + failed UI。  
3. 回归图片生成进度（图片本就有 clamp，行为应基本不变）。
