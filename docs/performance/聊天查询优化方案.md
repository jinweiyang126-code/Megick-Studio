# 聊天查询优化方案

本文档描述生产环境 `GET /api/chats/:id` 触发的 MySQL **Out of sort memory (1038)** 问题根因，以及全库类似「一次加载全量」风险的排查结论与分阶段改造设计。

---

## 1. 背景与问题

### 1.1 生产报错

```
GET /api/chats/:id?limit=10
PrismaClientUnknownRequestError: chatSession.findFirst()
MySQL 1038: Out of sort memory, consider increasing server sort buffer size
```

### 1.2 根因（已修复路径）

`ChatsService.detail()` 在带 `messageLimit` 时曾使用：

1. `count` 会话下全部消息数
2. `skip = total - limit`，再 `orderBy createdAt asc` + `skip` + `take`

MySQL 需对该会话**全部消息**升序排序后再跳过前面 N 行。消息量大且 `content`（TEXT）、`metadata`（JSON）较大时，排序缓冲区溢出。

**已落地修复（代码未合并到本文档时请以仓库为准）：**

- 有 `limit`：`orderBy desc` + `take N`，内存 `reverse()`
- 迁移：`(sessionId, createdAt)` 复合索引

### 1.3 仍存在的风险

| 路径 | 问题 |
|------|------|
| `GET /api/chats/:id` **无** `limit` | 仍加载会话全部消息 |
| `GET /api/admin/chats/:id` | Admin `detailAdmin()` 全量 `include.messages` |
| `POST /api/admin/templates/extract-from-chat` | 为找一条消息加载整段会话 |
| `GET /api/chats` | ~~用户全部未归档会话，且每会话 `jobs` 无 `take`~~ **已修复（P1-2）** |

---

## 2. 全库风险扫描摘要

### 2.1 高风险（与本次同类）

| 模块 | 接口 | 行为 |
|------|------|------|
| Chats | `GET /api/chats/:id`（无 limit） | 全量消息 + generationJob |
| Chats | `GET /api/admin/chats/:id` | 全量消息 + job + 元数据展开 |
| Templates | `POST /api/admin/templates/extract-from-chat` | `include.messages` 全量升序 |

### 2.2 中风险

| 模块 | 接口 | 行为 |
|------|------|------|
| Chats | `GET /api/chats` | 会话列表无分页；每会话 jobs 全量 |
| Generation | `GET /api/generation/jobs` | `offset` + `orderBy desc` 深分页 |
| Media Center | 列表 | `skip: offset` |
| Credits | `GET /api/users/me/credits/ledger` | `skip` 分页 |
| Admin | `GET /api/admin/dashboard` | 近 14 天 job/user 等 `findMany` 后在 Node 聚合 |

### 2.3 低风险

配置类与小表全量：AI Models、RBAC、OAuth、Site Settings、Navigation、Showcase Admin、Desktop Releases 等。Templates 公开列表内部 cap 80 条。

### 2.4 相对安全的模式

`GET /api/chats/:id/messages?before=&limit=` 使用 **游标 + desc + take**，无 `skip` 全量排序。

---

## 3. 改造目标

1. 消除 `chat_messages` 上的无界全量排序 / 全量加载
2. 控制 API 响应体与 Node 内存峰值
3. 保持 Studio 体验：首屏最近 N 条 + 向上翻页

---

## 4. P0：必须改

### 4.1 `GET /api/chats/:id` — 默认 limit

| 项 | 建议 |
|----|------|
| 无 `limit` | 默认 `limit=50`（常量如 `CHAT_DETAIL_DEFAULT_LIMIT`） |
| 上限 | `max=100`（与现有 controller 一致） |
| 全量 | 不对普通用户开放；长会话仅通过分页接口拼接 |
| 响应扩展 | `messagePagination: { total, hasMore, oldestCursor }` |

**兼容性：** 旧客户端无 `limit` 时从「全量」变为「最近 50 条」。Studio 主路径已用 `limit=10`；需同步改 `-studio-shared.tsx`、`session-media.ts` 等无 limit 调用。

**相关代码：**

- `apps/api/src/modules/chats/chats.service.ts` → `detail()`, `loadSessionMessages()`
- `apps/api/src/modules/chats/chats.controller.ts`
- `apps/web/src/routes/-studio-shared.tsx`
- `apps/web/src/megickcut/integration/session-media.ts`

### 4.2 `GET /api/admin/chats/:id` — 分页

```
GET /api/admin/chats/:id?limit=50&before=<iso>
```

- 首屏默认 `limit=50`，逻辑与用户侧 `paginateMessages` 一致（`desc` + `take` + `reverse`）
- 翻页：复用 `GET /api/chats/:id/messages` 语义，或新增 `GET /api/admin/chats/:id/messages`
- 会话元数据（user、title、counts）与 messages 分查，避免嵌套 `include` 全量

**相关代码：**

- `apps/api/src/modules/chats/chats.service.ts` → `detailAdmin()`
- `apps/web/src/routes/admin.user-chats.tsx`

### 4.3 `POST /api/admin/templates/extract-from-chat` — 定点查询

不再 `include` 全会话消息。按入参分支：

| 入参 | 查询 |
|------|------|
| `messageId` | `chatMessage.findFirst({ id, sessionId })` + job |
| `generationJobId` | `chatMessage.findFirst({ generationJobId, sessionId })` |
| 均无 | `findFirst`：`sessionId` + `role=assistant` + 有 job，`orderBy createdAt desc` |

补充 user 上下文：该 message 之前最近一条 `role=user`（`createdAt < ?`，`desc`，`take 1`）。

**相关代码：**

- `apps/api/src/modules/templates/templates.service.ts` → `extractFromChat()`

## 5. P1：应改

### 5.1 前端统一带 `limit`

| 文件 | 改为 |
|------|------|
| `apps/web/src/routes/-studio-shared.tsx`（约 882、1744、2112 行） | `?limit=10` 或共享常量 |
| `apps/web/src/megickcut/integration/session-media.ts` | 默认 `limit=50` 或按场景 10 |

建议封装 `fetchChatDetail(id, { limit })`，避免裸调 `/api/chats/${id}`。

### 5.2 `GET /api/chats` — 会话列表分页

```
GET /api/chats?page=1&pageSize=30
```

| 项 | 建议 |
|----|------|
| 默认 | `pageSize=30`，max 100 |
| `jobs` | `take: 5`（`orderBy createdAt desc`），够算 `mode` |
| `messages` | 保持 `take: 8`，仅 `metadata` |
| 响应 | `{ items, total, page, pageSize }` |

**兼容性：** 旧客户端若期望数组，可短期保留原形状 + `Deprecation`，或版本化 API。

**相关代码：**

- `apps/api/src/modules/chats/chats.service.ts` → `list()`
- `apps/web/src/routes/dashboard.chats.tsx`
- `apps/web/src/routes/-dashboard-shell.tsx`

### 5.3 索引补齐

```prisma
// chat_messages（P0 迁移）
@@index([sessionId, createdAt])

// chat_sessions（P1 建议）
@@index([userId, archived, updatedAt])

// generation_jobs（P1 建议，若 list 按 user 频繁查）
@@index([userId, createdAt])
```

迁移目录示例：`apps/api/prisma/migrations/20260706100000_chat_messages_session_created_at_idx/`

---

## 7. 验收标准

1. 含 5000+ 消息的会话：`GET /api/chats/:id?limit=10` 稳定可用，无 MySQL 1038
2. `extract-from-chat`（带 `messageId`）仅 2–3 条 SQL，无全会话 sort
3. 重度用户 `GET /api/chats` 响应体可控（分页后 < 500KB 量级）
4. 慢查询日志无 `chat_messages` 大 `filesort` + 大 `skip`
