# 管理后台审计日志接入方案

本文档描述 Megick Studio 开源版 **Admin Audit Log** 的现状、根因与分阶段接入设计。对应管理后台页面：**Admin → 审计日志**（`/admin/audit-log`）。

---

## 1. 背景与问题

### 1.1 现象

审计日志页可正常打开，但列表始终为空。

### 1.2 根因

| 组件 | 状态 |
|------|------|
| 数据库表 `admin_audit_logs` | ✅ 已存在 |
| 读取 API `GET /api/admin/audit-log` | ✅ 已实现 |
| 写入服务 `AdminAuditService.log()` | ✅ 已定义（`apps/api/src/modules/admin/admin-audit.service.ts`） |
| 各管理写操作调用写入 | ❌ **未接入** |

`AdminAuditService` 已在 `AdminModule` 中注册并 export，但全项目 **没有任何 Controller / Service 注入或调用**。管理后台的保存、删除、改配置等操作均不会写入审计表。

### 1.3 相关代码位置

| 路径 | 说明 |
|------|------|
| `apps/api/prisma/schema.prisma` → `AdminAuditLog` | 数据模型 |
| `apps/api/src/modules/admin/admin-audit.service.ts` | 写入服务 |
| `apps/api/src/modules/admin/admin.controller.ts` → `GET audit-log` | 分页查询 |
| `apps/web/src/routes/admin.audit-log.tsx` | 前端列表页（只读，无需改动即可展示数据） |

## 2. 数据模型

### 2.1 表结构 `admin_audit_logs`

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | cuid | 主键 |
| `adminId` | string? | 操作管理员用户 ID，可空（系统任务） |
| `action` | `AdminAction` | `CREATE` / `UPDATE` / `DELETE` / `EXEC` |
| `targetType` | string(64) | 业务对象类型，如 `user`、`ai_model`、`oss_config` |
| `targetId` | string(128)? | 对象 ID / code |
| `before` | Json? | 变更前快照（可选） |
| `after` | Json? | 变更后快照或请求摘要（可选） |
| `ip` | string(64)? | 客户端 IP |
| `userAgent` | string(512)? | User-Agent |
| `createdAt` | DateTime | 创建时间 |

索引：`adminId`、`targetType`。

### 2.2 `AdminAction` 语义建议

| 值 | 适用场景 |
|----|----------|
| `CREATE` | 新建资源（POST 创建实体） |
| `UPDATE` | 更新资源（PATCH / PUT / upsert） |
| `DELETE` | 删除资源 |
| `EXEC` | 非 CRUD 的管理动作（调积分、批量操作、测试连接、登录等） |

---

## 3. 设计目标

1. **快速可用**：用较少改动让审计页有记录。
2. **安全合规**：敏感字段脱敏，不记录明文密钥。
3. **可演进**：先全局兜底，再对关键接口补 `before` / `after`。
4. **不影响主流程**：审计写入失败不应阻断管理操作（建议 catch + logger）。

---

## 4. 总体方案：两阶段

```
阶段一（拦截器）          阶段二（显式记录）
/api/admin/** 写请求  →   关键 Service 补 before/after
     ↓                          ↓
AdminAuditService.log()   AdminAuditService.log()
     ↓                          ↓
admin_audit_logs          admin_audit_logs（更精细）
```

---

## 5. 阶段一：全局 Admin 写操作拦截器

### 5.1 范围

对路径匹配 `/api/admin/**` 的写请求统一记一条日志：

- `POST`、`PATCH`、`PUT`、`DELETE`

### 5.2 排除规则（避免刷屏）

| 规则 | 示例 |
|------|------|
| 所有 `GET` | `audit-log`、`dashboard`、`queues` |
| `*/test` 测试连接 | OSS / R2 / SMTP `POST .../test` |
| 可选：审计列表自身 | `GET /api/admin/audit-log`（已是 GET，天然排除） |

### 5.3 字段映射

| 审计字段 | 来源 |
|----------|------|
| `adminId` | `req.user.id`（须已登录且为 SUPER_ADMIN） |
| `action` | HTTP 方法 → `POST`=CREATE, `PATCH`/`PUT`=UPDATE, `DELETE`=DELETE；特殊路由可覆写为 EXEC |
| `targetType` | 从路径解析，见 §5.5 |
| `targetId` | 路径参数 `id`、`code` 等 |
| `after` | 请求 body（脱敏后），或响应体摘要（可选） |
| `ip` | `req.ip` 或 `x-forwarded-for` |
| `userAgent` | `req.headers['user-agent']` |

### 5.4 脱敏字段

写入 `before` / `after` 前，对以下 key（含嵌套）替换为 `__REDACTED__`：

- `accessKeySecret`、`secretAccessKey`、`apiKey`、`password`
- `clientSecret`、`accessKeySecretEnc`、`apiKeyEnc`
- `__KEEP_EXISTING__` 占位符可保留或记为 `[UNCHANGED]`

建议工具：`apps/api/src/common/utils/redact-secrets.ts`。

### 5.5 `targetType` 路径映射建议

| 路径前缀 | targetType |
|----------|------------|
| `/api/admin/users` | `user` |
| `/api/admin/rbac/roles` | `role` |
| `/api/admin/ai-models` | `ai_model` |
| `/api/admin/model-providers` | `model_provider` |
| `/api/admin/ai-image-edit-modes` | `ai_image_edit_mode` |
| `/api/admin/site-settings` | `site_setting` |
| `/api/admin/oauth-providers` | `oauth_provider` |
| `/api/admin/cloud-resources/oss-config` | `oss_config` |
| `/api/admin/cloud-resources/r2-config` | `r2_config` |
| `/api/admin/smtp` | `smtp_config` |
| `/api/admin/templates` | `prompt_template` |
| `/api/admin/showcase` | `showcase` |
| `/api/admin/navigation-menus` | `navigation_menu` |
| `/api/admin/desktop-updates` | `desktop_release` |
| `/api/admin/chats` | `chat_session` |
| `/api/admin/auth/login` | `admin_login`（action 建议 `EXEC`） |
| 其他 | 路径段拼接，如 `unknown_admin_route` |

### 5.6 建议文件结构

```
apps/api/src/common/
  interceptors/admin-audit.interceptor.ts
  utils/redact-secrets.ts

apps/api/src/modules/admin/
  admin-audit.service.ts          # 已有，可增加 logSafe() 包装
```

### 5.7 注册方式

任选其一：

- **全局**：`app.module.ts` 中 `APP_INTERCEPTOR` + 路径判断
- **模块级**：各 admin Controller 使用 `@UseInterceptors(AdminAuditInterceptor)`

推荐全局拦截器 + 路径前缀判断，维护成本最低。

### 5.8 错误处理

```typescript
// 伪代码：审计失败只打日志，不 throw
try {
  await this.audit.log({ ... });
} catch (err) {
  this.logger.warn(`Audit log failed: ${err}`);
}
```

## 6. 阶段二：关键操作显式记录

对安全与资金相关接口，在 **Service 层** 于变更前后各读一次状态，写入完整 `before` / `after`。

### 6.1 优先级 P0（必须先做）

| 模块 | 接口 | action | targetType | 备注 |
|------|------|--------|------------|------|
| 用户 | `PATCH /api/admin/users/:id/status` | UPDATE | `user` | 禁用/启用账号 |
| 用户 | `POST /api/admin/users/:id/credits/adjust` | EXEC | `credit_ledger` | 单人调积分 |
| 用户 | `POST /api/admin/users/credits/adjust-bulk` | EXEC | `credit_ledger` | 批量调积分 |
| RBAC | `POST/PATCH/DELETE /api/admin/rbac/roles` | CREATE/UPDATE/DELETE | `role` | 角色与权限 |

### 6.2 优先级 P1（配置类）

| 模块 | 接口 | action | targetType |
|------|------|--------|------------|
| AI 模型 | `POST/DELETE /api/admin/ai-models` | CREATE/DELETE | `ai_model` |
| Provider | `POST/DELETE /api/admin/model-providers` | CREATE/DELETE | `model_provider` |
| 图片编辑模式 | `POST/DELETE /api/admin/ai-image-edit-modes` | CREATE/DELETE | `ai_image_edit_mode` |
| 站点设置 | `POST/DELETE /api/admin/site-settings` | CREATE/DELETE | `site_setting` |
| OAuth | `POST /api/admin/oauth-providers` | UPDATE | `oauth_provider` |
| OSS | `POST /api/admin/cloud-resources/oss-config` | UPDATE | `oss_config` |
| R2 | `POST /api/admin/cloud-resources/r2-config` | UPDATE | `r2_config` |
| SMTP | `POST /api/admin/smtp` | UPDATE | `smtp_config` |

### 6.3 优先级 P2（内容与运营）

| 模块 | 接口 | action | targetType |
|------|------|--------|------------|
| 模板 | 模板 / 分类 CRUD | CREATE/UPDATE/DELETE | `prompt_template` / `template_category` |
| 首页展示 | showcase CRUD | CREATE/DELETE | `showcase` |
| 导航菜单 | navigation-menus CRUD | CREATE/DELETE | `navigation_menu` |
| 桌面更新 | desktop releases CRUD | CREATE/UPDATE/DELETE | `desktop_release` |
| 聊天审查 | admin chats 写操作（若有） | 视具体接口 | `chat_session` |

### 6.4 只读接口（阶段一/二均不记录写审计）

- `GET /api/admin/dashboard`
- `GET /api/admin/audit-log`
- `GET /api/admin/queues`
- `GET /api/admin/generation/jobs`（当前仅列表查询）

### 6.5 显式调用示例

```typescript
await this.audit.log({
  adminId: admin.id,
  action: "UPDATE",
  targetType: "user",
  targetId: userId,
  before: { status: previousStatus },
  after: { status: nextStatus },
  ip: req.ip,
  userAgent: req.headers["user-agent"],
});
```

## 8. 实施顺序

| 步骤 | 内容 | 预估 |
|------|------|------|
| 1 | `redact-secrets.ts` + `AdminAuditInterceptor` | 0.5～1 天 |

## 9. 验证清单

1. 以 SUPER_ADMIN 登录管理后台。
2. 执行一次写操作（如保存 OSS 配置、新建 AI 模型）。
3. 打开 **Admin → 审计日志**，应出现新记录。
4. 记录应包含：时间、管理员邮箱、`action`、`targetType`、可选 `targetId`。
5. 数据库抽查：

```sql
SELECT id, action, targetType, targetId, adminId, createdAt
FROM admin_audit_logs
ORDER BY createdAt DESC
LIMIT 20;
```

6. 确认 `after` 中无明文 `accessKeySecret`、`apiKey` 等。

---

## 10. 与 RBAC 的关系

当前开源版管理接口统一使用 `@Roles("SUPER_ADMIN")`，**未**按 `permissions` 表做细粒度鉴权。审计日志记录的是「谁（SUPER_ADMIN）做了什么」，与后续完整 RBAC 演进兼容；阶段二完成后，可为非 SUPER_ADMIN 的自定义角色预留相同 `AdminAuditService` 接口。
